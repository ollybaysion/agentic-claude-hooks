# keyword-docs-add

`context` 모듈의 keyword-docs 계열 인덱스에 **문서 하나를 등록하는 절차**.
등록되면 그 키워드가 프롬프트에 등장할 때마다 문서가 자동 주입된다 — 인덱스는
매 턴 다시 읽히므로 **append 즉시 적용**(리로드 불필요). 인덱스 수정은 반드시
**사용자 승인 후에만** 한다.

인자 `[문서경로]`: 등록할 문서 (프로젝트 상대경로). 없으면 물어본다.

## 1. 문서 확인

- 인자 경로의 문서를 Read로 읽고 요지를 파악한다. **파일이 없으면 종료**하고
  경로를 확인해 달라고 안내한다.
- 문서가 아주 크면(>10k자) 참고: 주입은 앞 `maxCharsEach`(기본 1200자)만
  들어가므로, **문서 서두에 핵심 요약이 오도록** 권고한다 (강제는 아님).

## 2. 카테고리(인스턴스) 선택 — 하드코딩하지 말고 실제 설정을 본다

`<project>/.claude/context.json`의 `providers`를 읽어 **이 프로젝트에 켜져 있는
keyword-docs 계열 인스턴스**와 각자의 인덱스 경로(`params.index` 오버라이드 포함)를
파악한다. 기본 인스턴스와 기본 인덱스 경로:

| 인스턴스 | 기본 인덱스 | 용도 |
| --- | --- | --- |
| `keyword-docs` | `.claude/context-docs.json` | 범용 |
| `msg-format` | `.claude/context-docs.msg-format.json` | 설비 커맨드 → 메시지 포맷 |
| `db-schema` | `.claude/context-docs.db-schema.json` | DB/테이블명 → 스키마 문서 |
| `domain-docs` | `.claude/context-docs.domain.json` | 도메인 용어 → 개념 문서 |

문서 내용에서 카테고리를 **추정해 제안하되, 확정은 사용자**가 한다
(AskUserQuestion — 추정 카테고리를 첫 옵션으로). 켜져 있지 않은 인스턴스를
고르면: 인덱스에 추가는 하되 "이 인스턴스가 `.claude/context.json`에 꺼져 있어
주입되지 않는다"고 알리고 켤지 물어본다.

## 3. 트리거 키워드 제안

문서에서 트리거 후보를 뽑는다 (제목, 고유 용어, 식별자, 반복 등장 개념). 규칙:

- **ASCII 소문자만** — 한글 단독 키워드는 토크나이저(`[a-z0-9_]+`)에 안 걸린다.
  사용자가 프롬프트에 섞어 쓰는 영어 용어를 노린다.
- 단어 모드(기본)는 **복수형 허용**(`migration` ↔ `migrations`), 밑줄 식별자는
  한 토큰(`cmd_start_lot`). 여러 단어 phrase(공백 포함)는 부분일치.
- **너무 범용인 단어는 경고**하고 빼거나 phrase로 좁힌다 — 아무 프롬프트에나
  발화하는 키워드는 오탐 = 토큰 낭비다 (`code`, `file`, `git` 단독 등).
- 3~6개가 적당. 제안 목록을 보여주고 사용자가 고르거나 수정하게 한다.

## 4. 충돌 검사 — 존재하는 모든 인덱스 파일 전부

먼저 매치된 항목이 뒤 항목을 가리므로(엔진이 인덱스 순서로 스캔, 같은 키워드가
다른 문서에 있으면 그 인스턴스 안에서 섀도잉), **같은 키워드가 두 곳에 있으면
안 된다.** 검사 범위:

- 위 표의 기본 인덱스 경로 4개 중 존재하는 것 전부
- `.claude/context.json`의 `params.index`로 지정된 커스텀 인덱스 전부

제안 키워드 각각을 기존 모든 항목의 `keywords`와 대조한다(소문자 비교). 겹치면:
어느 파일/문서와 겹치는지 보여주고, **더 좁은 키워드/phrase로 교체**를 제안한다.
(서로 다른 인스턴스 간 중복은 기술적으론 각자 주입돼 이중 주입이 된다 — 역시
피한다.)

## 5. precision 확정

- 키워드가 좁고 확실(고유 식별자·도메인 용어) → `precision` 생략 (기본 1, 전문 주입).
- 키워드가 넓거나 확신이 낮음 → **`0.5` 제안** — 한 줄 포인터만 주입(~20토큰),
  관련이 진짜면 모델이 직접 Read. §3에서 "범용이라 경고했지만 그래도 넣고 싶은"
  키워드는 이쪽이 기본값이다.

## 6. 승인 후 append

1. 최종 항목을 보여주고 승인받는다:

   ```json
   { "keywords": ["tsum", "total sum"], "path": "docs/domain/tsum.md", "precision": 1 }
   ```

2. 승인되면 해당 인덱스 파일에 항목을 append한다. 파일이 없으면 `[항목]`으로
   생성한다. **prettier 포맷을 유지**한다(lint 훅이 검사한다) — 수정 후
   `prettier --write <인덱스>`를 돌리는 게 확실하다.
3. **승인 없는 수정 금지.** 항목 외 기존 내용은 절대 건드리지 않는다.

## 7. 마무리 안내

- 추가 내역 요약: 어느 인스턴스 인덱스에, 어떤 키워드로, precision 얼마로.
- "다음 턴부터 즉시 적용" + "몇 주 뒤 `/keyword-docs-prune`으로 이 키워드의
  오탐 여부를 데이터로 확인하라"고 안내한다.

## 한계

- 키워드 품질은 결국 사람 판단이다 — 이 스킬은 후보 추출·충돌 검사·규칙 검증까지만
  하고, 확정은 사용자가 한다 (prune과 같은 철학).
- 인스턴스 간 이중 등록(같은 문서를 두 카테고리에)은 막지 않는다 — 키워드만 안
  겹치면 유효한 사용법일 수 있다. 단 그 사실을 사용자에게 알린다.
