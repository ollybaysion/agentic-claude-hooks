목차

    시퀀스 패턴 견본

    html-doc 스킬 · patterns/sequence.html — 이 파일 자체가 복사 원본이다 (시퀀스 패턴 CSS:START…END 블록 + 아래 데모 마크업)

    1. 언제 쓰나

    두 주체가 시간 순서대로 주고받는 흐름을 설명할 때 — API 왕복, 훅 ↔ 하네스 이벤트, 미러 ↔ 서버 동기화, 프로토콜 핸드셰이크. 시간이 위에서 아래로 흐르고, 가로 화살표 하나 = 실제로 오간 메시지 하나다. 단순 구조·관계 표현은 pre.diagram이 더 싸다 — 왕복의 내용물까지 보여줘야 할 때만 이 패턴을 쓴다.

    구성 블록 6종: actors(주체 카드, 상단 고정) · wire(화살표+라벨) · payload(실린 내용: 요지 스택 + 접힌 원문) · local(한쪽 내부 처리, 점선) · edge(시작/끝 경계) · keys(마무리 요점 카드).

    2. 데모 — 클라이언트 ↔ 서버 왕복 두 번

    클라이언트
    요청을 시작하는 쪽 · 재시도 정책 소유
    서버
    무상태 처리 · 저장은 DB에 위임
    시작 사용자가 버튼 클릭 — 여기서부터 왕복이 시작된다
    #1 · 요청 · POST /things

    생성 요청 — 본문에 이름과 옵션

    인증 토큰은 헤더에 — 본문에는 자격증명이 없다.

    client {"name":"예시","opts":{"dry_run":false}} 필드 검증 전 — 서버를 신뢰하지 않는 값
    원문 그대로 보기 
    POST /things HTTP/1.1
Authorization: Bearer …
Content-Type: application/json

{"name":"예시","opts":{"dry_run":false}}
    서버 내부 — 와이어 밖에서 일어나는 일

    검증 → 저장 → 식별자 발급

    1. 스키마 검증 (name 필수, 63자 제한)
    2. INSERT INTO things … — bind 변수로만

    클라이언트는 이 과정을 보지 못한다 — 보이는 건 응답뿐.

    #1 · 응답 · 201 Created

    생성 결과 — 서버가 발급한 id

    server {"id":"th_01","name":"예시"} id는 서버 발급 — 이후 모든 요청의 짝맞춤 키
    #2 · 요청 · GET /things/th_01

    방금 만든 리소스 재조회

    읽기 요청엔 본문이 없다 — 식별자는 경로에.

    #2 · 응답 · 200 OK

    저장된 그대로 반환 — 왕복 종료 판정

    server {"id":"th_01","name":"예시","created":"2026-07-19"}
    끝 클라이언트가 화면을 갱신 — 시퀀스는 여기서 닫힌다

    화살표 = 실제 메시지

    추상 관계가 아니라 실제로 오간 것만 wire로 그린다. 내부 처리는 local 점선 박스로 분리 — "선 위"와 "선 밖"의 구분이 이 패턴의 정보량이다.

    요지는 펼치고 원문은 접는다

    카드에는 해석된 요지(스택·주석), 원문 페이로드는 details.rawbox에 보존 — 가독성과 검증 가능성을 동시에.

    3. 사용법 (모델용 절차)

    1. 시퀀스 패턴 CSS:START…END 블록을 문서 <style> 끝에 통째로 복사한다. 액터 색을 주제에 맞게 바꿔도 되지만 라이트/다크 쌍은 반드시 유지한다.
    2. 시퀀스 패턴 마크업:START…END 블록을 복사해 내용을 교체한다. A = 왼쪽(시작하는 쪽), B = 오른쪽. 방향은 wire.to-b(A→B) / wire.to-a(B→A).
    3. 규칙: 다이어그램 내부에서는 h2/h3를 쓰지 않는다(.card-title 사용 — heading id·TOC 규율과 충돌). 색이 의미를 가지면 본문에 범례를 먼저 쓴다. 원문 데이터가 있으면 요지로 재작성하되 원문을 rawbox에 보존한다.
    4. 렌더 후 check.mjs는 평소대로 — 이 패턴은 전부 토큰 참조라 추가로 지킬 것이 없다.
